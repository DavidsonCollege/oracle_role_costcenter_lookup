PROJECT NOTES — Oracle Access Request Lookup
=============================================
Reference notes for future development on this or similar projects.


APPLICATION OVERVIEW
--------------------
Electron desktop app (Mac + Windows) that queries Oracle HCM Cloud via
REST API and SOAP to look up worker positions, assigned security roles,
auto-provisioning rules, and cost center manager information.

Files:
  main.js       — Electron main process; all API calls live here
  preload.js    — Context bridge exposing IPC to the renderer
  index.html    — Single-page UI markup and styles (no inline scripts)
  renderer.js   — All UI logic (loaded via <script src="renderer.js">)
  package.json  — Build config for electron-builder
  assets/       — App icons (icon.icns for Mac, icon.ico for Windows)
  build-date.json        — Written at build time by prebuild:mac/prebuild:win scripts
  entitlements.mac.plist — Required for Mac hardened runtime
  .github/workflows/build.yml — GitHub Actions release pipeline
  control-budget-app/    — Standalone stripped-down app for Control Budget report only


ORACLE HCM REST API
-------------------
Base path: {baseUrl}/hcmRestApi/resources/latest/

Authentication: HTTP Basic Auth
  - Header: Authorization: Basic <base64(username:password)>
  - Build token: Buffer.from(`${username}:${password}`).toString('base64')

Key endpoints used:
  /workers
    - Query by PersonNumber: q=PersonNumber=12345
    - Expand nested data: expand=workRelationships,workRelationships.assignments
    - Returns: PersonId, DisplayName, workRelationships[].assignments[].PositionCode
    - Assignment priority: prefer PrimaryFlag=true AND AssignmentStatusType=ACTIVE

  /positions
    - Query by position code: q=PositionCode='ABC-123'  (single quotes required)
    - Single-quote escaping in query values: replace ' with ''
    - Returns: Name (also try PositionName, DisplayName as fallbacks)

  /publicWorkers
    - Fallback endpoint if /workers access is restricted
    - Supports: FirstName, LastName, DisplayName, KnownAs, FullName queries

  /userAccounts
    - Query by PersonId: q=PersonId=12345
    - Returns account with a 'links' array containing a 'self' href

  /userAccountRoles  (child resource)
    - Access via: {selfHref}/child/userAccountRoles?limit=500
    - IMPORTANT: Using expand on the parent only returns ~25 rows (default page).
      Always follow the self link and query the child resource directly with
      an explicit limit to get all roles.
    - Returns: RoleCode, CreationDate, LastUpdateDate
    - Only DAV_SEC_* and *REPORTS* role codes are returned — other roles
      (HR, payroll, etc.) are filtered out as irrelevant to access reviews.


ORACLE HCM SOAP API (PublicReportService)
-----------------------------------------
Endpoint: {baseUrl}/xmlpserver/services/PublicReportService
SOAPAction header value: "runReport"
Content-Type: text/xml; charset=utf-8

Envelope structure:
  <runReport xmlns="http://xmlns.oracle.com/oxp/service/PublicReportService">
    <reportRequest>
      <reportAbsolutePath>/Custom/path/to/Report.xdo</reportAbsolutePath>
      <attributeFormat>xlsx</attributeFormat>
      <sizeOfDataChunkDownload>-1</sizeOfDataChunkDownload>
        <!-- -1 = return entire report in one response, no streaming/pagination -->
    </reportRequest>
    <userID>username</userID>
    <password>password</password>
  </runReport>

IMPORTANT — fault handling:
  Oracle BI Publisher returns HTTP 200 even when a report fails (e.g. wrong
  path, permission denied). Always check for a SOAP <Fault> element in the
  response body BEFORE checking response.ok. The real error is in the fault,
  not the HTTP status code.

Report path gotcha: the .xdo extension is required in reportAbsolutePath.
  Omitting it returns a permission error (not "not found"), which is misleading.

Response parsing:
  - Report data is returned as base64 in the <reportBytes> element
  - Decode: Buffer.from(b64, 'base64')
  - Parse xlsx in-memory with SheetJS: XLSX.read(buffer, { type: 'buffer' })
  - Convert to 2D array: XLSX.utils.sheet_to_json(sheet, { header: 1, defval: '' })

Report row structure (varies):
  - Some reports: row 0 = title, row 1 = column headers, row 2+ = data
  - Other reports: row 0 = column headers, row 1+ = data
  - Robust approach: scan rows with Array.findIndex() to locate the header row
    by looking for a known column name rather than assuming a fixed row index

Report paths used in this project:
  Cost Center Manager:      /Custom/Jessamyn/CostCenterManager/DeptCostCenterMgr.xdo
  Auto-Provisioning Rules:  /Custom/Jessamyn/Auto-Provisioning Rules/Auto-Provisioning Rules.xdo
  Cost Center Hierarchy:    /Custom/Jessamyn/AccountHierarchy/CostCenterHierarchy.xdo

Known report column names (upper-cased after normalisation):
  Cost Center report:
    COST CENTER, COST_CENTER        — cost center number
    DEPARTMENT NAME, DEPT NAME      — department name
    COST CENTER MANAGER, CC MANAGER — manager name
    CC MANAGER STATUS, MANAGER STATUS
    COST_CENTER_MGR_EMAIL           — manager email
    COST_CENTER_MGR_NUM             — manager person number

  Auto-Provisioning Rules report:
    MAPPING_NAME, DEPARTMENT, POSITION_CODE, JOB, ROLE

  Cost Center Hierarchy report:
    HIERARCHY_PATH (col B) — contains parent cost center codes (e.g. C480);
                             no surrounding underscores, matched as substring
    CHILD_VALUE    (col E) — child cost center value; falls back to col index 4
                             if header not found
    Filter logic: rows where HIERARCHY_PATH includes the target code AND
    CHILD_VALUE does not start with 'C'. Values starting with 'C' are
    intermediate parent nodes in the Oracle hierarchy tree, not real cost centers.


ROLE CODE LINK BEHAVIOUR
------------------------
In the Assigned Roles table, DAV_SEC_* role codes are rendered as clickable links:

  DAV_SEC_C<digits>  (e.g. DAV_SEC_C480)
    — Hierarchical cost center group. Clicking runs the CostCenterHierarchy
      SOAP report and merges all matching child cost centers into the Cost
      Center Report field (deduplicates with any values already present).

  DAV_SEC_<digits>  (e.g. DAV_SEC_1234)
    — Direct cost center code. Clicking appends the segment after DAV_SEC_
      to the Cost Center Report field if not already present.

  All other role codes are shown as plain text.


ELECTRON ARCHITECTURE
---------------------
- contextIsolation: true, nodeIntegration: false (secure defaults)
- All Node.js / API work happens in main.js (main process)
- Renderer (index.html) communicates only via ipcRenderer.invoke()
- preload.js exposes a named API object: window.hcmAPI.{methodName}

IPC pattern used:
  Main:    ipcMain.handle('channel:name', async (_event, params) => { ... })
  Preload: methodName: (params) => ipcRenderer.invoke('channel:name', params)
  UI:      const result = await window.hcmAPI.methodName(params)

Return shape convention:
  Success: { ok: true, ...data }
  Failure: { ok: false, error: 'message' }
  This lets the UI check result.ok before accessing data fields.

Secondary/optional data (roles, auto-prov rules) uses a nested try/catch so
failures are surfaced as error strings rather than crashing the whole lookup.


SETTINGS PERSISTENCE
--------------------
Non-sensitive settings (baseUrl, username) are saved to a JSON file:
  Location: app.getPath('userData') + '/settings.json'
  Mac:      ~/Library/Application Support/<appName>/settings.json
  Windows:  C:\Users\<user>\AppData\Roaming\<appName>\settings.json

Pattern:
  Read:  JSON.parse(fs.readFileSync(settingsPath, 'utf8'))  — wrap in try/catch
  Write: fs.writeFileSync(settingsPath, JSON.stringify(data), 'utf8')
  Passwords are intentionally NOT persisted.


ELECTRON-BUILDER PACKAGING
---------------------------
package.json "build" key controls packaging.

Mac:     "target": [
           { "target": "dmg", "arch": ["universal"] },
           { "target": "zip", "arch": ["universal"] }
         ]
         Both targets are required: DMG for initial install, ZIP for auto-updates.
         universal = single artifact that runs on both Intel and Apple Silicon.
         hardenedRuntime: true, notarize: true — required for signing/notarization.
         Icon:  assets/icon.icns

Windows: "target": [{ "target": "nsis", "arch": ["x64"] }]
         NSIS produces a standard .exe installer
         oneClick: false — lets users choose install directory
         Icon:  assets/icon.ico  (must be at least 256x256)

Build commands (local only — do not use with --publish in production):
  npm run build:mac   — Mac only
  npm run build:win   — Windows only
  npm run build       — both

Cross-platform builds: handled automatically by GitHub Actions (see DEPLOYMENT
section below). No Wine or Windows machine required.

Files array must explicitly include all runtime dependencies:
  "files": ["main.js", "preload.js", "renderer.js", "index.html",
            "build-date.json", "assets/**",
            "node_modules/xlsx/**", "node_modules/electron-updater/**"]
  electron-builder does NOT auto-include node_modules unless specified here.
  Forgetting renderer.js breaks the UI in packaged builds.


DEPLOYMENT & AUTO-UPDATES
--------------------------
Releases are built and published via GitHub Actions (.github/workflows/build.yml).
The workflow triggers on any v* tag push and runs three jobs:
  1. create-release (ubuntu-latest) — pre-creates the GitHub Release
  2. build on macos-latest  → universal DMG + ZIP, signed and notarized
  3. build on windows-latest → NSIS .exe
Jobs 2 and 3 run in parallel after job 1 completes. Pre-creating the release
in a separate job prevents both build jobs from racing to create it (which
causes a 422 error for whichever finishes second).

Both platforms are published as assets on the GitHub Release using GH_TOKEN.

To release a new version:
  1. Bump "version" in package.json (semver: patch for fixes, minor for features)
  2. Commit and push main FIRST:
       git add package.json && git commit -m "v1.x.x"
       git push origin main
  3. Then create and push the tag:
       git tag v1.x.x && git push origin v1.x.x

  IMPORTANT: push main before the tag. GitHub uses the workflow file from the
  commit the tag points to. If the tag is pushed at the same time as workflow
  changes, it may point to the old workflow.

  Never use --publish always locally after pushing the tag — the release already
  exists on GitHub and the local build will fail with 422.

First-time setup: repo → Settings → Actions → General →
  set Workflow permissions to "Read and write permissions".

Auto-update behaviour (electron-updater, runtime dependency):
  Both platforms: autoDownload=true. The updater downloads silently in the
  background, then fires the 'update-downloaded' event.

  Windows: full auto-update via checkForUpdatesAndNotify() — user is prompted
           to restart when the download is complete.

  Mac:     autoDownload=true + update-downloaded event → Electron dialog.showMessageBox
           with "Restart Now" / "Later" buttons → autoUpdater.quitAndInstall()
           on confirm. Full auto-update works because builds are code-signed.

  Mac auto-update requirements — ALL of these must be true:
    1. Code-signed and notarized build (Developer ID Application certificate)
    2. ZIP target in Mac build config (electron-updater uses the zip, not the DMG)
    3. autoDownload = true set before calling checkForUpdates()
    4. update-downloaded listener registered before calling checkForUpdates()

  Do NOT use system notifications for Mac update alerts — the app may not appear
  in macOS Notification Center, so notifications are silently dropped.
  Use dialog.showMessageBox on the update-downloaded event instead.

  Update check runs inside an app.isPackaged guard so it is skipped during
  development (npm start).

Code signing — Mac (configured):
  Signed and notarized via GitHub Actions using a Developer ID Application cert.
  Certificate and credentials are stored as GitHub Secrets:
    APPLE_CERT               — base64-encoded .p12 (run: base64 -i cert.p12 | pbcopy)
    APPLE_CERT_PASSWORD      — export password (type manually — do NOT paste;
                               pasting can introduce hidden characters)
    APPLE_ID                 — Apple ID email
    APPLE_APP_SPECIFIC_PASSWORD — from appleid.apple.com
    APPLE_TEAM_ID            — 10-character team ID from developer.apple.com

  Certificate gotchas:
    - Developer ID Application certs can only be created by the Account Holder
      on the Apple Developer account. Admin role is not sufficient.
    - CSR must be generated on the same Mac where the .p12 will be exported.
      If the .cer is downloaded on a different Mac, the private key is absent
      and import fails with error -25294.
    - Wrong password errors despite correct password: likely hidden characters
      in the GitHub secret — delete and retype manually.

  Supporting files:
    entitlements.mac.plist — enables JIT, unsigned executable memory, network
                             client access (required for hardened runtime)

Code signing — Windows (not yet configured):
  SmartScreen shows "unknown publisher" warning on install; users click
  More info → Run anyway. Requires a certificate from DigiCert or Sectigo.


UI PATTERNS
-----------
- index.html — markup and styles only; no inline scripts or event handlers
- renderer.js — all UI logic; loaded via <script src="renderer.js">
  (kept separate from HTML for CSP compliance)
- CSS uses system font stack: -apple-system, BlinkMacSystemFont, "Segoe UI"
- Oracle red brand color: #c74634
- Result areas are hidden (display:none) until populated, then toggled with
  CSS classes: .result.success / .result.error / .result.info
- Loading state on buttons: add class 'loading' — CSS hides label, shows spinner
- HTML injection prevention: always pass user-visible strings through esc()
  function that escapes &, <, >, " — required on every innerHTML write to
  prevent XSS from API response content
- Enter key on any field triggers the relevant action via keydown listeners
- Button clicks wired with addEventListener — no onclick attributes in HTML
- Dynamic rows (candidates, role links) use data-* attributes + delegated
  event listeners — no inline onclick
- Buttons are disabled when required fields are empty (updateButtonStates()).
  Call updateButtonStates() after any programmatic field value change, not
  just after user input events.
